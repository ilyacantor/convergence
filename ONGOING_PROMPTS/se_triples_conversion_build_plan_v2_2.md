# **0\. What Changed from v1 and v2.0**

## **0.1 Current State: The SE Pipeline Works**

The SE E2E pipeline is confirmed working as of March 29, 2026\. It runs 6 steps, orchestrated by Console’s pipeline\_orchestrator.py:

| Step | What It Does | Calls |
| :---- | :---- | :---- |
| 1\. Farm Snapshot | Generates an enterprise snapshot for the tenant. Produces snapshot\_id, entity\_id, tenant\_name. | Farm :8003/api/snapshots |
| 2\. AOD Discovery | Takes snapshot\_id from Step 1\. Discovers and classifies assets. Returns run\_id \+ asset counts. | AOD :8001/api/runs/from-farm |
| 3\. AOD→AAM Handoff | Exports discovered candidates from AOD to AAM using run\_id from Step 2\. | AOD :8001/api/handoff/aam/export |
| 4\. AAM Inference | Infers pipe definitions from candidates. Returns pipes\_created count. | AAM :8002/api/aam/infer |
| 5\. DCL Ingest (Farm Financials) | Orchestrator calls Farm’s manifest-intake to generate financial triples, then pushes them to DCL’s HTTP ingest endpoint. Returns rows\_generated and rows\_accepted. | Farm :8003/api/farm/manifest-intake → DCL :8004/api/dcl/ingest |
| 6\. Pipeline Complete | Summary. Counts successes/failures/skips, totals elapsed time. | — |

Baseline performance: 22 seconds total, 5/5 steps succeeded, 23,393 financial triples accepted into DCL.

The critical detail: only Step 5 writes triples to DCL. AOD and AAM do their jobs in the pipeline chain (discover, handoff, infer), but their output feeds the next stage — not DCL. All 23K triples in DCL are financial triples generated by Farm. AOD and AAM produce zero triples today.

## **0.2 What v1 Proposed**

v1 of this plan proposed two work packages (WP-AOD, WP-AAM) that would add triple conversion to AOD and AAM, so they also write discovery.\* and mapping.\* triples to the semantic store. The idea: Maestra could then answer questions like “what systems were discovered?” and “how are systems connected?” from triples, not just from AOD/AAM’s internal data models.

v1 also instructed AOD and AAM to write triples directly to Postgres via execute\_values, bypassing DCL’s HTTP ingest endpoint entirely.

## **0.3 What v2.0 Corrected**

v2.0 corrected the write path: if AOD and AAM write triples, they must go through DCL’s HTTP ingest endpoint (same path Farm Financials uses), not direct PG. This aligns with the pipeline identity architecture and provides the paved road to WP-8 (AAM Blueprint production ingest pipeline).

## **0.4 What v2.1 Changes: AOD/AAM Triple Writes Are Deferred**

**The AOD and AAM triple conversion work packages (WP-AOD, WP-AAM) are deferred.**

The SE pipeline works today without AOD or AAM writing triples. Their value in the SE chain is in the pipeline flow: AOD discovers, hands off to AAM, AAM infers pipes, the orchestrator dispatches Farm to generate financials, and Farm’s output goes to DCL. That chain delivers financial context. AOD and AAM contribute by feeding the chain, not by writing their own triples.

Adding discovery.\* and mapping.\* triples to DCL is a feature that enables richer Maestra queries. It is not required for the SE pipeline to function, and it is not on the critical path for Convergence, production deployment, or the WP-8 transition.

## **0.5 What v2.2 Corrects: ME Write Path and SE/ME Routing**

v2.1 §3 (SE and ME Are Separate Pipelines) described the ME pipeline as “Farm (per entity) → DCL (per entity) → COFA → Verify” and listed the ME write path as “POST /api/dcl/ingest-triples (direct triple push).” This was wrong. DCL is SE-only. ME triples never write to DCL’s semantic_triples table.

Corrected ME pipeline: Farm (per entity) → Convergence (per entity) → COFA → Verify. ME financial triples write to Convergence’s convergence_triples table via the Convergence backend (port 8010). COFA output also writes to convergence_triples. Convergence reads DCL-owned tables (SELECT only) for cross-reference. If ME data appears in DCL’s semantic_triples, that is a pipeline routing bug.

Governing doc references updated: convergence_MA_spec_v7.4.1, AOS_MASTER_RACI_v8.3.

## **0.6 When to Revisit**

Revisit WP-AOD and WP-AAM triple conversion when:

* Maestra needs to answer discovery or connectivity questions from triples rather than from AOD/AAM’s internal APIs.

* NLQ needs to query across discovery, mapping, and financial domains in a single query.

* WP-8 (production ingest pipeline) is built and provides a universal ingest endpoint.

When that work is picked up, the rules established in v2.0 still apply:

* AOD and AAM must push triples through DCL’s HTTP ingest endpoint. No direct PG writes.

* The orchestrator coordinates the push (same as Step 5: orchestrator calls the module, then pushes result to DCL). Modules do not call DCL directly.

* Triples carry the stage’s namespaced ID as run\_id (aod\_discovery\_id, aam\_inference\_id) per pipeline\_identity\_architecture\_v1.

* Conversion functions live in each module’s repo (AOD owns discovery.\* conversion, AAM owns mapping.\* conversion) per RACI v8.3.

## **0.7 Existing Direct PG Writes Are Tech Debt**

AOD and AAM currently have triple conversion code that writes directly to Postgres:

* AOD: \~/code/aod/src/aod/converters/triple\_converter.py — writes via asyncpg, bypasses DCL.

* AAM: \~/code/aam/app/converters/triple\_converter.py — writes via psycopg2 execute\_values(), bypasses DCL.

These are artifacts of the v1 build plan. They bypass DCL’s HTTP ingest, violate the pipeline identity architecture (no dcl\_ingest\_id issued), and are the reason stale ME data (meridian, cascadia, combined) ended up in DCL’s semantic\_triples table alongside SE data. These code paths should not be invoked by the SE pipeline. When the deferred work is picked up, they must be migrated to the DCL HTTP path, not extended.

# **2\. The EAV Data Model**

The DCL triple store uses an EAV (Entity-Attribute-Value) schema. Every fact in AOS is stored as a single row with this structure:

| Column | EAV Role | What It Represents | Example |
| :---- | :---- | :---- | :---- |
| entity\_id | Entity (scope) | Who this fact belongs to | BlueLogic-NEQ8, VeloHub-19JC |
| concept | Entity (type) | Business object, dot-separated: domain.object\_type | compensation.base, revenue.total, ebitda.bridge |
| property | Attribute | Specific attribute of the business object | annual\_amount, vendor, modality |
| value | Value | The data | $125,000, Salesforce, API |
| period | Temporal key | When this fact applies (null if atemporal) | 2024-Q4, 2025-01 |
| source\_system | Provenance | Which module produced this triple | Farm |
| source\_field | Provenance | Original field name in the source | base\_salary, revenue\_amount |
| confidence\_score | Provenance | Confidence (0.0–1.0) | 0.92, 0.75 |
| confidence\_tier | Provenance | Human-readable tier | HIGH, MEDIUM, LOW |
| pipe\_id | Provenance | Which data pipe produced this | farm\_financials-financials |
| run\_id | Provenance | Which execution run produced this | Stage’s namespaced ID per identity spec |
| created\_at | Provenance | When this triple was written | 2026-03-29T15:26:00Z |

Today, all triples in the SE store are financial triples from Farm (compensation.\*, revenue.\*, ebitda.\*, service\_catalog.\*, etc.). The concept column’s first segment is the domain, used by Maestra for retrieval filtering and NLQ for query routing.

# **3\. SE and ME Are Separate Pipelines**

AOD and AAM are SE-only stages. They do not run in ME mode.

|  | SE Pipeline | ME Pipeline |
| :---- | :---- | :---- |
| Stages | Farm Snapshot → AOD → Handoff → AAM → Farm Financials → DCL Ingest | Farm (per entity) → Convergence (per entity) → COFA → Verify |
| Repo | Console \+ Platform \+ AOD \+ AAM \+ Farm \+ DCL | Convergence (own repo, own backend, port 8010/3010) |
| Write target | DCL’s semantic\_triples via POST /api/dcl/ingest-triples | Convergence’s convergence\_triples (internal write). DCL never touched. |
| Entity model | One entity, fixed at tenant creation (e.g., BlueLogic-NEQ8) | Multiple entities from engagement config (e.g., meridian, cascadia) |
| COFA | Does not exist | Unifies chart of accounts across entities |

Do not mix their plumbing. ME data never writes to DCL’s semantic\_triples table. If ME data (meridian, cascadia, combined, UUID-entity rows) appears in DCL’s semantic\_triples table, that is a pipeline routing bug.

| AOD Object | Concept Prefix | Properties to Convert |

Hard constraints when built:

* Orchestrator pushes triples to DCL via HTTP. AOD does not call DCL directly. AOD does not write to PG directly.

* run\_id value \= aod\_discovery\_id per identity spec.

* Conversion function is a separate, testable module in the AOD repo.

* Triple write is additive. Does not replace the AAM handoff.
